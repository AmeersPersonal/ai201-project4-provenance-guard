# Provenance Guard

A small backend a creative platform can call to guess whether a piece of writing
was made by a person or generated by AI. It returns a verdict, a confidence
score that's honest about how sure it is, and a plain-language label to show
readers. Creators can appeal a verdict, the submission endpoint is rate limited,
and every decision is written to an audit log.

The point isn't to be a perfect detector (nobody has one). It's to be useful and
honest: a 0.51 call should look very different to a reader than a 0.95 one, and a
writer who gets flagged should have a way to push back.

## Quick start

```bash
# install
pip install -r requirements.txt

# optional: add a Groq API key for the LLM signal (see note below).
# edit .env and set GROQ_API_KEY=gsk_...

# put a few sample decisions in the log so GET /log has data
python seed.py

# run it (http://localhost:5000)
python src/main.py
```

### About the Groq key

The strongest signal, `llm_judge`, calls Groq. With a key it does most of the
work and confidence gets sharper. Without one the service still runs on the
three text statistics signals and just hands back more "uncertain" verdicts,
which is the honest thing to do when the best signal is missing. Every number in
this README was captured without a key, so you can reproduce it as-is.

## API

### POST /submit

Classify some content.

```bash
curl -X POST http://localhost:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"content": "Your poem or paragraph here...", "content_type": "text"}'
```

`content_type` defaults to `text`. The other options are `image_description` and
`metadata`. A response looks like this (signal details trimmed):

```json
{
  "content_id": "86deab82acfa",
  "content_type": "text",
  "result": "AI",
  "ai_probability": 0.877,
  "confidence": 0.732,
  "label": "🤖 Likely AI-generated. We're fairly confident this was made with a generative AI tool (73% confidence). This is an automated guess, not proof, and the creator can appeal it.",
  "signals": [
    {"name": "burstiness", "weight": 0.2, "score": 1.0, "available": true},
    {"name": "lexical_diversity", "weight": 0.15, "score": 1.0, "available": true},
    {"name": "ai_cliche", "weight": 0.15, "score": 1.0, "available": true}
  ],
  "status": "classified"
}
```

`content_id` is what you pass to `/appeal`, and it's how the decision shows up in
`/log`.

### POST /appeal

Contest a decision. The appeal gets logged next to the original decision and the
decision's status flips to `under_review`.

```bash
curl -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "86deab82acfa", "reasoning": "I wrote this by hand as satire."}'
```

### GET /log

Returns the whole audit log: every decision, newest first, with any appeals
attached.

### GET /

Basic info about the service.

## How detection works

The pipeline runs four signals that look at different things, so a piece has to
look AI in more than one way before the score gets confident. Each signal returns
a number from 0 (looks human) to 1 (looks AI), or nothing if there isn't enough
text to judge.

| Signal | What it looks at | Weight | Where it's weak |
|--------|------------------|:------:|-----------------|
| `llm_judge` | A Groq model reads the text and estimates how likely it is AI. This is the only signal that understands meaning. | 0.50 | Costs an API call; can be fooled by edited AI text; skipped entirely with no key. |
| `burstiness` | How much sentence length varies. People mix long and short sentences; AI tends to even out. | 0.20 | A human who writes short, even sentences (lists, minimalist prose) reads as AI. |
| `lexical_diversity` | Type-token ratio, i.e. how repetitive the vocabulary is. | 0.15 | Very length-sensitive and noisy; the weakest of the four. |
| `ai_cliche` | How often known LLM phrases show up ("delve", "rich tapestry", "it is important to note") plus em-dash density. | 0.15 | A human quoting or parodying that style trips it; a careful AI writer avoids it. |

The four scores get combined into one `ai_probability` as a weighted average,
and the weights are renormalized over whatever signals were available. If
`ai_probability >= 0.5` the result is `AI`, otherwise `Human`.

For the `metadata` content type the text signals don't apply, so it uses
`llm_judge` plus a `metadata_signature` check that looks for generator tags
(like `"generator": "Midjourney v6"`) and zeroed-out timestamps.

## Confidence

A binary "AI or human" would hide how shaky a lot of these calls are, so the more
important number is confidence. It combines two things: how far the probability
is from the 0.5 line, and how much the signals agree with each other.

```
p_ai          = weighted average of the available signal scores
result        = AI if p_ai >= 0.5 else Human
boundary_term = 2 * abs(p_ai - 0.5)            # 0 right at 0.5, 1 at the ends
agreement     = clamp(1 - 2 * stdev(scores))   # 1 when the signals agree
confidence    = clamp(0.5 * boundary_term + 0.5 * agreement)
```

So a verdict is only confident when the probability is clearly to one side and
the signals back each other up. If either is shaky, confidence drops and the
reader gets the "uncertain" label instead.

### Checking the scores actually mean something

`python calibrate.py` runs the pipeline over a labelled set of human and AI
samples and checks two things (offline, no key):

```
=== CLAIM 1: SEPARATION ===
  human mean p_ai = 0.067
  ai    mean p_ai = 0.665
  separation = +0.597   (PASS: want clearly positive)

=== CLAIM 2: MONOTONIC CONFIDENCE ===
  ensemble p_ai -> confidence (signals held unanimous):
    p_ai=0.50 -> confidence=0.500
    p_ai=0.55 -> confidence=0.550
    p_ai=0.60 -> confidence=0.600
    p_ai=0.70 -> confidence=0.700
    p_ai=0.80 -> confidence=0.800
    p_ai=0.95 -> confidence=0.950
  PASS: confidence rises with distance from the 0.5 boundary.
```

The first check shows the scores track reality: the AI samples land about 0.6
higher than the human ones, so the ranking isn't noise. The second shows
confidence climbs as the call gets less borderline, which is the whole reason a
0.51 reads differently from a 0.95.

Two example submissions with clearly different confidence:

- `python seed.py` sample "Morning light. The kettle clicks off..." comes back
  `Human`, `ai_probability` 0.028, `confidence` 0.939 (high).
- The buzzword paragraph ("In today's fast-paced world, it is important to
  note...") comes back `AI` at `ai_probability` 0.652 but only `confidence`
  0.206, because the signals disagree, so it gets the uncertain label.

## Transparency labels

The label is the bit a reader would actually see. There are three of them,
chosen by confidence (0.70 and up counts as confident). `{N}` is the confidence
as a whole-number percent.

| When | Label text |
|------|------------|
| `result` is AI and `confidence >= 0.70` | `🤖 Likely AI-generated. We're fairly confident this was made with a generative AI tool ({N}% confidence). This is an automated guess, not proof, and the creator can appeal it.` |
| `result` is Human and `confidence >= 0.70` | `✍️ Likely human-written. We're fairly confident a person wrote this ({N}% confidence). This is an automated guess, not proof.` |
| `confidence < 0.70` | `❓ Origin unclear. We couldn't confidently tell whether a person or an AI tool produced this ({N}% confidence), so treat the result as inconclusive.` |

Real ones from the seeded log:

- `🤖 Likely AI-generated. We're fairly confident this was made with a generative AI tool (73% confidence). This is an automated guess, not proof, and the creator can appeal it.`
- `✍️ Likely human-written. We're fairly confident a person wrote this (94% confidence). This is an automated guess, not proof.`
- `❓ Origin unclear. We couldn't confidently tell whether a person or an AI tool produced this (21% confidence), so treat the result as inconclusive.`

Each one says what it thinks, gives a number, and makes clear it's a guess rather
than an accusation. On a writing platform, calling a real person's work "AI" is
the worse mistake, so the wording leans cautious and always points to the appeal.

## Appeals

A creator who thinks a verdict is wrong sends their `content_id` and a short
reason to `/appeal`. The service:

1. Saves their reasoning in the `appeals` table.
2. Links it to the original decision, so both show up together in `/log`.
3. Sets the decision's status to `under_review`.

It doesn't re-run detection automatically; an appeal just flags the decision for
a human to look at. The response echoes the updated decision:

```json
{
  "appeal_id": "d5d51107763a",
  "content_id": "54b494c1b89a",
  "status": "under_review",
  "message": "Appeal recorded. The original decision is now under review."
}
```

## Rate limiting

`POST /submit` is limited to **10 per minute and 100 per day** per IP
(Flask-Limiter, in-memory store).

The reasoning: a submission can trigger a paid LLM call, so the per-minute cap is
mainly about cost and abuse. Ten a minute is plenty for a real integration
(nobody is hand-submitting work faster than that) but it kills a script trying to
hammer the endpoint. The daily cap puts a ceiling on what one client can cost
even if it stays slow. Only `/submit` is limited; `/log`, `/appeal` and `/` are
not. Over the limit returns 429. Quick check:

```
request 1..9  -> HTTP 201
request 10    -> HTTP 429
```

## Stretch features

**Ensemble of 3+ signals.** The pipeline runs four signals with fixed, written-down
weights (0.50 / 0.20 / 0.15 / 0.15) blended by a renormalized weighted average.
See the detection and confidence sections above.

**A second content type.** `/submit` branches on `content_type`. Besides `text`
it handles `image_description` (same text pipeline, LLM prompt tuned for
captions) and `metadata` (a JSON blob of creation attributes scored by the
`metadata_signature` signal, which flags tool tags and zeroed timestamps). The
seeded Midjourney metadata entry above is an example.

## Known limitations

The text statistics are easy to fool one at a time, which is on purpose: they're
meant to back up the LLM signal, not stand alone. The clearest failure is short,
plain human writing with even sentence lengths. The `burstiness` signal reads
that uniformity as AI, and without a key there's no semantic signal to overrule
it, so honest human prose can land in the uncertain band. Adding a Groq key fixes
most of this by putting the deciding weight back on the signal that actually reads
the text.

To reproduce everything: `python seed.py`, then `python calibrate.py`, then
`python src/main.py` and the curl commands above.
